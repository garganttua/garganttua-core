================================================================================

[LAYOUT]

[VISUEL]
Graphe de dependances interactif (librairie : D3.js, vis.js, ou Cytoscape.js).
Les modules sont des noeuds colores par couche :
- Vert clair   : Foundation (commons, dsl, supply, lifecycle, mutex)
- Bleu         : Infrastructure (reflection, runtime-reflection, condition, execution, crypto, configuration)
- Violet       : Framework (injection, runtime, mapper, expression, bootstrap)
- Orange       : Application (script, console, workflow)
- Gris         : Build Tools (annotation-processor, script-maven-plugin)
- Jaune        : Integration (bindings: spring, reflections, mutex-redis, native)
- Rouge        : AOT (aot-commons, aot-reflection, aot-annotation-scanner, aot-annotation-processor, aot-maven-plugin)

Interactions :
- Hover sur un noeud : le noeud s'agrandit, ses dependances directes
  s'illuminent, les autres se grissent. Un tooltip affiche le nom complet
  et la description one-liner.
- Click sur un noeud : ouvre un panneau lateral ou une modale avec la
  description complete du module et un lien vers son README.
- Filtre par couche : boutons radio au-dessus du graphe pour isoler
  une couche ("Foundation", "Infrastructure", etc.)

Alternative statique (fallback/mobile) : image SVG du graphe Mermaid stylisee,
avec une legende des couleurs par couche en dessous.

[TEXTE]

    Une architecture en 7 couches

    Chaque module respecte une hierarchie stricte : les couches basses
    ne dependent jamais des couches hautes. Le resultat : un graphe
    de dependances previsible, sans surprises et sans cycles.

[LEGENDE sous le graphe]

    Foundation          Interfaces, builders, suppliers, lifecycle
    Infrastructure      Reflexion (IClass, force access), conditions, execution, crypto, configuration multi-format
    Framework           Injection, runtime, mapping, expressions, bootstrap
    Application         Scripts, workflows, console
    Build Tools         Annotation processing, plugins Maven
    Integration         Spring, Reflections, Redis, GraalVM native
    AOT                 Compilation anticipee, generation de code, plugin Maven

--- Carte 1 : Injection ---

    garganttua-injection                                [badge: Framework]

    Conteneur d'injection de dependances leger et compatible JSR-330.
    Scopes singleton et prototype, injection de proprietes, contextes
    enfants, detection de cycles, et instanciation zero-reflexion
    optionnelle via binders compiles.

    IDiContextBuilder builder = DiContextBuilder.create()
        .withBean(UserService.class)
        .withBean(UserRepository.class)
        .withProperty("app.name", "MyApp")
        .build();

    context.onStart();
    UserService service = context.getBean(UserService.class);

--- Carte 2 : Expression ---

    garganttua-expression                               [badge: Framework]

    Langage d'expressions base sur ANTLR4 avec evaluation type-safe.
    Appels de fonctions, invocations de methodes, constructeurs,
    references de variables, et types generiques. L'evaluation produit
    des ISupplier<T> pour le calcul differe. Fonction cast() pour le
    transtypage avec resolution dynamique du type de retour. Support
    des fonctions utilisateur via DynamicFunctionNode.

    // Appel de fonction avec arguments
    context.expression("add(8, add(42, 30))")

    // Transtypage avec resolution dynamique du type retour
    context.expression("cast(String.class, @myVar)")

    // Reference de variable + constructeur
    context.expression(":(String.class, @myVar)")

--- Carte 3 : Script ---

    garganttua-script                                [badge: Application]

    Moteur de scripting complet avec sa propre grammaire ANTLR4,
    fonctions utilisateur, execution conditionnelle via if(),
    gestion des erreurs, et synchronisation via mutex. Extension
    .gs, arguments positionnels, et fat JAR executable.
    Console REPL dans un module dedie.

    // Fonctions utilisateur avec scope isole
    validate = (data) => (
        result <- checkFormat(@data)
    )
    output <- validate(@input)

    // Execution conditionnelle
    result <- if(equals(@mode, "prod"), (
        data <- fetchProd()
    ), (
        data <- fetchDev()
    ))

    // Gestion d'erreur immediate
    result <- parse_json(@0) ! Exception.class => log("parse failed")

--- Carte 4 : Workflow ---

    garganttua-workflow                              [badge: Application]

    DSL fluide pour orchestrer des pipelines multi-etapes.
    Genere automatiquement du code Garganttua Script a partir
    de l'API builder. Les scripts inline sont isoles dans des
    groupes (...) pour eviter les collisions de fonctions entre
    stages. Execution conditionnelle via if(), filtrage de
    stages au runtime, et support inline/include.

    Workflow workflow = WorkflowBuilder.create()
        .name("deploy-pipeline")
        .stage("build")
            .script("scripts/compile.gs")
            .script("scripts/test.gs")
            .up()
        .stage("deploy")
            .when("equals(@env, \"prod\")")
            .script("scripts/deploy.gs")
            .up()
        .build();

    workflow.execute(input);

--- Carte 5 : Configuration ---

    garganttua-configuration                      [badge: Infrastructure]

    Chargement de configuration multi-format (JSON, YAML, XML,
    TOML, Properties) avec population automatique de builders.
    Mapping intelligent des cles (camelCase, kebab-case, snake_case),
    conversion de types, et integration DI.

    var populator = ConfigurationBuilder.builder()
        .withMappingStrategy("SMART")
        .strict(true)
        .build();

    populator.populate(serverBuilder,
        new FileConfigurationSource("config.yaml"));

--- Carte 6 : Reflection ---

    garganttua-reflection                        [badge: Infrastructure]

    Facade de reflexion pluggable avec IReflection, IClass<T>, et
    providers priorises. Binders type-safe (IMethodBinder, IFieldBinder,
    IConstructorBinder) avec evaluation lazy via ISupplier. Navigation
    par adresse objet (dot-notation) et resolution de methodes. Acces
    force aux membres prives via parametre force. Abstraction IExecutable
    commune pour methodes et constructeurs.

    // Construire la facade composite avec priorites
    var reflection = ReflectionBuilder.builder()
        .withProvider(new RuntimeReflectionProvider(), 10)
        .withScanner(new MyAnnotationScanner(), 10)
        .build();

    // Acces force a un champ prive via ObjectAddress
    reflection.forceGetFieldValue(obj, new ObjectAddress("inner.secret"));

    // FieldAccessor avec dot-notation
    var accessor = new FieldAccessor<>(ownerType, "address.city");
    var value = accessor.supply();

SECTION 7 - ZERO REFLEXION AU RUNTIME (differenciateur technique)
================================================================================

[LAYOUT]
Section split : texte a gauche (60%), visuel a droite (40%).
Fond #0D1117.

[VISUEL - cote droit]
Schema anime ou illustration montrant la transformation :

    Phase de compilation                    Phase d'execution
    +----------------------------+          +----------------------------+
    |  @Expression("add")        |          |  addBinder.invoke(a, b)    |
    |  static int add(int, int)  |   --->   |  = direct method call      |
    |                            |          |  NO Method.invoke()        |
    +----------------------------+          +----------------------------+
    |  Annotation Processor      |          |  DirectBinderRegistry      |
    |  genere AddMethodBinder    |          |  charge le binder          |
    +----------------------------+          +----------------------------+

Style : deux blocs cote a cote relies par une grosse fleche.
Le bloc gauche est en couleur "compilation" (bleu), le droit en "runtime" (vert).
Animation : la fleche pulse et les blocs apparaissent sequentiellement.

[TEXTE - cote gauche]

    Elimination de la reflexion sur les chemins critiques

    La plupart des frameworks Java utilisent Method.invoke() et
    Constructor.newInstance() au runtime, avec un cout significatif
    en performance et en compatibilite native.

    Garganttua Core prend une approche differente : un annotation
    processor genere des classes de binders directs a la compilation.
    Le resultat :

    - Les fonctions @Expression appellent directement la methode
      Java cible, sans passer par Method.invoke()

    - Les beans @Singleton et @Prototype sont instancies par
      new ClassName(...), pas par Constructor.newInstance()

    - La configuration GraalVM Native Image est simplifiee :
      @Reflected remplace @Native pour declarer les usages,
      et moins de metadata de reflexion a declarer

    - Le demarrage est plus rapide : pas de scanning au classpath,
      les index META-INF/garganttua/index/ sont pre-generes

    Activable par defaut. Desactivable via :
    -Dgarganttua.direct.binders=false

[TEXTE]

--- Carte 1 ---
    [icone: terminal >_]

    Outils ligne de commande

    Combinez garganttua-script et GraalVM Native Image pour creer
    des CLI tools Java qui demarrent en millisecondes. Le REPL
    interactif accelere le prototypage. Le fat JAR executable
    se deploie sans dependances externes.

    Modules concernes : script, console, expression, bootstrap

--- Carte 2 ---
    [icone: hexagones connectes]

    Microservices legers

    Utilisez garganttua-injection comme alternative a Spring
    quand vous avez besoin d'un conteneur DI sans la surface
    d'API d'un framework full-stack. Injection JSR-330,
    lifecycle management, et configuration multi-format
    dans une empreinte memoire reduite.

    Modules concernes : injection, configuration, lifecycle, bootstrap

--- Carte 3 ---
    [icone: fleches traversant des boites]

    Pipelines de traitement

    Orchestrez des workflows multi-etapes avec gestion d'erreurs,
    fallbacks, et execution conditionnelle via if(). Le module
    workflow genere du code script optimise a partir de votre API
    builder, avec isolation automatique des fonctions entre stages.
    Filtrage de stages au runtime, variables partagees entre etapes,
    et execution reproductible.

    Modules concernes : workflow, script, runtime, execution, condition
